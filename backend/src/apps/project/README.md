# Project Service (production/project-management)

Сервис управления съёмочными проектами: проекты, смены, участники смен (с интервалами занятости), документы смен (план/сценарий), а также заявки на использование ресурсов (камера/звук/свет/реквизит) на конкретный интервал времени.

Репозиторий является частью дипломного проекта и описывает (и/или реализует) **project-service** — сервис “организации” и “планирования” внутри production‑workflow.

## Контекст и границы ответственности

### За что отвечает project-service (источник правды)
- **Проекты**: создание проекта, описание, владелец/создатель.
- **Состав проекта**: участники проекта и их роли (режиссёр, реквизитор, камера, звук, свет, актёр).
- **Смены**: создание/редактирование смены, утверждение (workflow статусов).
- **Участие в сменах**: кто участвует и **в какой интервал времени** (time_from/time_to), статусы подтверждения/резерва.
- **Документы смены**: загрузка/просмотр/выгрузка файлов (план, сценарий) в единой модели документов.
- **Заявки на ресурсы**: запросы на использование предметов (оборудование/реквизит) на интервал времени, статусы, связь с внешним резервом.
- **Оркестрация уведомлений**: формирование событий/команд на рассылку в notification-service.

### За что НЕ отвечает project-service
- CRUD пользователя, профиль пользователя.
- CRUD предметов пользователя (оборудование/реквизит) и их окна доступности.
- Фактическая проверка занятости, разруливание конфликтов по времени, нарезка окон — это зона `user-service`.

## Интеграции (внешние сервисы)

### User Service
Источник правды для:
- ресурсов пользователя (оборудование/реквизит),
- окон доступности пользователя и ресурсов,
- резервирования интервала внутри существующего free‑окна (`free -> reserved`).

Project-service использует user-service для:
- проверки существования пользователя,
- получения доступных ресурсов/окон,
- создания резерва времени (для пользователя или конкретного ресурса),
- отмены резерва (нужно согласовать API).

### Notification Service
Сервис рассылок (SMTP). Project-service публикует события “что случилось”:
- приглашение в проект,
- добавление/изменения в смене,
- загрузка документов,
- запрос/подтверждение ресурса и т.д.

## Ключевые сущности домена
- **Project** — съёмочный проект.
- **Project Member** — пользователь в проекте + роль.
- **Shift** — смена (интервал, статус).
- **Shift Participant** — участие пользователя в смене с интервалом (time_from/time_to) и статусом (приглашён/подтверждён/зарезервирован).
- **Document** — файл смены (PLAN / SCENARIO) в единой таблице.
- **Resource Request** — запрос на использование ресурса (camera/sound/light/requisite/…) на интервал времени + связь с внешним резервом.

## Примечание про транспорт интеграций
Термин “event” в описании диплома используется как “обращение в другой сервис”.
Технически это может быть:
- HTTP/gRPC (синхронный запрос/ответ),
- request/reply поверх брокера сообщений,
- асинхронные события (fire-and-forget) для уведомлений.

Внутри домена важно различать:
- **команды/запросы** (ожидаем ответ, меняем статус),
- **события** (факт, уведомление других сервисов).

## Предлагаемая структура репозитория

- `docs/` — спецификации и диаграммы
- `src/` — код сервиса
- `migrations/` — миграции БД
- `.env.example` — переменные окружения
- `docker-compose.yml` — инфраструктура для dev (Postgres, RabbitMQ, MinIO/S3), если используется

## Roadmap (MVP)
1) Проекты + участники проекта + роли  
2) Смены + участники смен (по времени) + утверждение смен  
3) Документы смен (план/сценарий)  
4) Заявки на ресурсы на интервал + интеграция резерва с user-service  
5) Уведомления по событиям (notification-service)
